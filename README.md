# Credit Card Fraud Detection

A machine learning project to detect fraudulent credit card transactions using the [Credit Card Fraud Detection dataset](https://www.kaggle.com/datasets/mlg-ulb/creditcardfraud) from Kaggle. 

## Problem Statement

Can I build a model that reliably catches fraudulent transactions despite them being extremely rare, without generating excessive false alarms?

The dataset contains 284,807 European credit card transactions over two days, with only 492 labeled as fraud (about 0.17% of the data). Features V1 to V28 are PCA-transformed and anonymized. Time and Amount are the only raw fields, alongside the target Class (0 = normal, 1 = fraud).

Accuracy is not a meaningful metric here. A model that predicts "Not Fraud" for every transaction scores about 99.8% accuracy while catching zero fraud. This project focuses on the precision-recall tradeoff and the $F_2$-score instead: missing fraud (false negatives) costs the bank actual money, while flagging too many legitimate transactions (false positives) creates unnecessary manual review work and erodes customer trust.

## Approach & Methodology

1. **Exploratory Data Analysis (EDA)**: Analyzed class distributions, transaction amounts, time patterns, and feature correlations with the target.
2. **Robust Preprocessing**: Checked for nulls and duplicates. Applied a `RobustScaler` to the Time and Amount columns (chosen over `StandardScaler` for its lower sensitivity to outlier transactions), with V1-V28 passed through unchanged. All preprocessing is wrapped in a single `ColumnTransformer` and reused across every model via a scikit-learn `Pipeline` to prevent data leakage.
3. **Data Splitting (60/20/20)**: To properly train, tune, and evaluate the models, the dataset was split into three distinct sets using random stratified splitting (`stratify=y`):
   * **Testing Set (20%)**: Split off first to act as the final, completely unseen evaluation set.
   * **Validation Set (20%)**: Used specifically for tuning model hyperparameters and decision thresholds.
   * **Training Set (60%)**: Used to train the initial models.
   * *Note:* Stratification guarantees that the exact same proportion of fraudulent transactions (~0.17%) is preserved across all three sets, preventing a scenario where one set accidentally contains zero fraud cases.
4. **Addressing Class Imbalance**: Experimented with Random Undersampling and SMOTE (Synthetic Minority Over-sampling Technique) to give the models a balanced environment to learn from.
5. **Threshold Tuning**: Since the default 0.50 decision threshold is often too conservative for fraud detection, I explicitly tuned the decision thresholds of the top models using the validation set. The goal was to optimize for Recall and the $F_2$-Score to aggressively catch hackers.
6. **Explainability (SHAP)**: Used SHAP (SHapley Additive exPlanations) values and waterfall plots to break open the "black box" of the tree models and identify the strongest indicators of a stolen credit card (e.g., V14, V10, V17).

## Results & Model Comparison

After systematically preprocessing the dataset, diagnosing overfitting, and manually optimizing decision thresholds, the following table summarizes the complete performance metrics across all tested models on the unseen test set:

| Model | Accuracy | Precision (Fraud) | Recall (Fraud) | F1 (Fraud) | F2 (Fraud) |
| ------ | ------ | ------ | ------ | ------ | ------ |
| Logistic Regression + class weight | 0.9985 | 0.53 | 0.88 | 0.66 | 0.7773 |
| Decision Tree | 0.9992 | 0.74 | 0.80 | 0.77 | 0.7847 |
| KNN | 0.9994 | 0.91 | 0.74 | 0.82 | 0.7733 |
| Naive Bayes | 0.9762 | 0.06 | 0.85 | 0.11 | 0.2287 |
| XGBoost (default threshold) | 0.9924 | 0.17 | 0.89 | 0.29 | 0.4828 |
| XGBoost (validation-tuned threshold = 0.90) | 1.0000 | 0.62 | 0.85 | 0.72 | 0.7890 |
| XGBoost (SMOTE default) | 0.9900 | 0.11 | 0.90 | 0.19 | 0.3601 |
| XGBoost (SMOTE + tuned threshold = 0.90) | 1.0000 | 0.57 | 0.87 | 0.69 | 0.7841 |
| Random Forest (default threshold) | 0.9995 | 0.94 | 0.77 | 0.84 | 0.7945 |
| Random Forest (undersampled) | 0.9800 | 0.07 | 0.90 | 0.13 | 0.2731 |
| Random Forest (SMOTE default) | 0.9995 | 0.86 | 0.83 | 0.84 | 0.8333 |
| Random Forest (SMOTE + tuned threshold = 0.40) | 1.0000 | 0.82 | 0.84 | 0.83 | 0.8333 |

### Key Takeaways & Final Recommendation

* **The Accuracy Trap:** Notice how Naive Bayes has a 97.6% accuracy, yet its Precision is an abysmal 0.06 (meaning 94% of its alerts are false alarms). This highlights exactly why Accuracy was ignored in favor of the $F_2$-score and Recall.
* **XGBoost's Overfitting & Sensitivity:** While XGBoost is exceptionally powerful, it proved highly unstable when exposed to the synthetic noise generated by SMOTE, suffering a severe drop in Precision (down to 0.11) at default thresholds. It also required heavy regularization to avoid memorizing the training set.
* **The Winning Trade-off (Random Forest 0.12):** The tuned Random Forest model (Threshold = 0.12, no SMOTE) is my recommended model for deployment. By manually lowering the decision threshold, I intentionally sacrificed a bit of precision (dropping from 0.94 to 0.69) to gain a massive operational security boost (Recall jumping from 0.77 to 0.87). This model catches 87% of all unseen future fraud while maintaining a manageable false positive rate for the investigation team.

## Project Structure

```text
.
├── notebook.ipynb              # Full analysis: EDA, preprocessing, modeling, evaluation, and SHAP
├── app.py                      # Streamlit interactive UI/dashboard
├── fraud_model.pkl             # Saved model: a single Pipeline containing both preprocessing and the trained Random Forest
├── sample_transactions.csv     # Synthetic sample file for trying the batch upload demo
├── requirements.txt            # Python dependencies
└── README.md
```

The saved model is a single scikit-learn `Pipeline` object, combining the `RobustScaler` preprocessing step and the trained `RandomForestClassifier` together. This means the Streamlit app only needs to load one file and can pass raw transaction data (Time, Amount, V1-V28) directly to it, with no manual feature scaling required in the application code.

## Running the Project

**1. Install dependencies:**
```bash
pip install -r requirements.txt
```

**2. Reproduce the Analysis & Model Training:**
To view the Exploratory Data Analysis, preprocessing steps, and model evaluations, run the Jupyter Notebook:
```bash
jupyter notebook notebook.ipynb
```

**3. Run the Interactive Dashboard:**
To launch the Streamlit UI where you can test the model with synthetic transactions and visually adjust the confidence threshold:
```bash
streamlit run app.py
```
